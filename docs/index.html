Fase 2 · Enumeración de Directorios

Gobuster — Descubrimiento de Rutas

Comando
gobuster dir \ -u http://10.40.37.240:5173 \ -w /usr/share/wordlists/dirbuster/\ directory-list-2.3-medium.txt \ -o intel/dirs-frontend.txt
gobuster.png
Hallazgo técnico importante
Wildcard Response detectada
El servidor devuelve 200 OK (521 bytes) para cualquier ruta inexistente (UUID aleatorio generado por gobuster). Comportamiento esperado de SPA con client-side routing fallback: todas las rutas devuelven index.html al router del frontend.
¿Vulnerabilidad?
No es una vulnerabilidad. Invalida el escaneo de directorios en modo default. Requiere ajuste técnico.
Ajuste aplicado
Relanzar con --exclude-length 521 para filtrar wildcard y obtener resultados de rutas reales.

Fase 3

Threat Modeling — Modelado de Amenazas

F-3 EN PROGRESO
Vectores de ataque identificados
Sin API Gateway / Reverse Proxy
Microservicios expuestos directamente en LAN sin capa de protección intermedia.
Comunicación sin cifrar (HTTP)
Todo el tráfico entre cliente y microservicios en texto plano — vulnerable a sniffing en LAN.
PostgreSQL expuesto (5432)
Puerto de base de datos accesible desde LAN sin firewall intermedio.
Vite preview en producción
Servidor de desarrollo Vite no es seguro para producción.
Escenarios de ataque
Escenario A — auth-service (crítico)
Fuerza bruta de credenciales · JWT weak signing/verification · Bypass de autenticación
Escenario B — Base de datos
PostgreSQL sin autenticación o credenciales débiles · SQL Injection a través de APIs
Escenario C — Lateral Movement
Desde un contenedor comprometido, pivotar hacia otros. Escalada de privilegios Docker.
Actualización basada en evidencia real
CORS más restrictivo de lo esperado
auth-service y patient-service tienen whitelist activa — hipótesis de "CORS permisivo" revisada.

Fase 7 · Reporting

Recomendaciones de Remediación

Remediaciones prioritarias
🔴 Alta prioridad
• Poner PostgreSQL detrás de firewall interno (no exponer 5432 a LAN)
• Implementar API Gateway (Kong, Nginx) como punto único de entrada
• Habilitar TLS/HTTPS en todos los servicios
🟠 Media prioridad
• Migrar frontend a servidor de producción (Nginx/Apache) — retirar Vite preview
• Agregar helmet a Express para ocultar X-Powered-By
• Implementar rate limiting en auth-service (express-rate-limit)
🟡 Baja prioridad
• Auditar políticas CORS en admin, appointment y medical-service
• Verificar gestión segura de JWT secrets (rotación, expiración corta)
• Revisar variables de entorno y secretos en Docker Compose
Consideraciones éticas
Autorización previa
Nunca ejecutar pruebas sin autorización escrita del propietario del sistema.
Datos PHI/PII
Si se accede a datos de pacientes, documentar el acceso pero NO copiar datos fuera del entorno de prueba.
Sin DoS
Evitar pruebas que puedan causar denegación de servicio — respetar acuerdos de alcance.
Limpieza post-prueba
Eliminar cualquier backdoor, persistencia o artefacto dejado durante la explotación.
